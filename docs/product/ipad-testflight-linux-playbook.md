# Linux to iPad TestFlight Playbook

Status: active path for teams developing on Linux without a physical Mac.

## Goal

Ship a private iPad beta through TestFlight while keeping day-to-day development on Linux.

## Delivery model

1. Build and test web app on Linux.
2. Package iOS shell with Capacitor.
3. Use GitHub Actions `macos-latest` runners for Xcode archive/sign/upload.
4. Distribute via TestFlight Internal Testers first.

## Prerequisites

- Apple Developer Program membership.
- App Store Connect app record created.
- Bundle ID reserved for iOS app (example: `com.academicos.planner`).
- Hosted HTTPS URL for production-like web experience (used by `CAP_SERVER_URL`).
- Node.js `>=22` for local Capacitor CLI commands.

## Required GitHub secrets

- `APPLE_TEAM_ID`
- `APPSTORE_CONNECT_KEY_ID`
- `APPSTORE_CONNECT_ISSUER_ID`
- `APPSTORE_CONNECT_API_KEY_BASE64` (base64-encoded `.p8` content)
- `IOS_CERTIFICATE_P12_BASE64` (distribution certificate in `.p12`, base64-encoded)
- `IOS_CERTIFICATE_PASSWORD`
- `IOS_PROVISIONING_PROFILE_BASE64` (App Store profile, base64-encoded)
- `CAP_SERVER_URL` (example: `https://planner.example.com`)

## Local workflow (Linux)

```bash
pnpm install
pnpm --filter @planner/web build
pnpm --filter @planner/web cap:add:ios      # first time only
pnpm --filter @planner/web cap:sync:ios
pnpm --filter @planner/web test:e2e:ipad
```

Notes:
- `cap:add:ios` only needs to run when `apps/web/ios` does not exist.
- iOS compilation/signing is done in GitHub Actions macOS, not Linux.
- `test:e2e:ipad` validates critical routes in iPad viewport emulation; still run a real iPad Safari pass before closing B1.
- If local Node is `<22`, `cap:sync:ios` will fail. Use GitHub macOS workflow as the canonical signing path.

## iPad web smoke pass criteria

Complete all rows in [`ipad-beta-acceptance-checklist.md`](ipad-beta-acceptance-checklist.md) with evidence:
- Dashboard/nav usability in portrait and landscape.
- Resources upload -> indexing -> search roundtrip.
- Search + AI ask response sanity.
- Notes/notebooks touch input usability.
- Manifest/icon endpoint checks (`/manifest.webmanifest`, `/icon`, `/apple-icon`).

Triggers:
- manual dispatch
- tag push matching `ios-v*`

Pipeline stages:
1. Validate required secrets.
2. Build web app.
3. Ensure iOS platform exists and sync Capacitor.
4. Import signing cert/profile.
5. Archive via `xcodebuild`.
6. Export IPA.
7. Upload to TestFlight with `altool`.

Real-run readiness requirements before dispatch/tag:
- Apple signing checklist complete.
- GitHub secrets checklist complete.
- `CAP_SERVER_URL` points to reachable HTTPS environment.
- `apps/web/ios/App` exists or can be generated by `cap:add:ios`.
- Workflow Node version must remain `22` or newer for Capacitor CLI compatibility.

## TestFlight rollout protocol

1. Start with Internal Testers only.
2. Validate install, login/auth, upload flow, search flow, and core navigation on iPad.
3. Promote to External Testers only after internal pass.

Required evidence to mark rollout ready:
- `ios-testflight.yml` run URL (green).
- TestFlight build number visible in App Store Connect.
- At least one successful internal iPad install.
- Completed acceptance checklist with tester/device/date fields.

## Known constraints

- VirtualBox-only iOS signing/submission is not a reliable release path.
- App Store/TestFlight signing still requires Apple toolchain on macOS (handled by CI runners).
